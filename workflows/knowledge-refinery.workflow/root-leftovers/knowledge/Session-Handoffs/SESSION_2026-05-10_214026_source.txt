Claude responded: Good question, and it's the right question to ask before building more on top.
Good question, and it's the right question to ask before building more on top. Let me answer the memory mechanism directly because that's what you're actually asking, then we'll talk about the architecture.

How does it remember 50 things without getting infinitely large?

It doesn't store 50 pictures as 50 memories. It compresses. Here's the mechanism:

Each snapshot goes through a pipeline: screenshot → local vision model (moondream on your NAS) → text description of what's on screen → embedding vector. That embedding is a fixed-size numerical fingerprint — maybe 384 or 768 numbers, regardless of whether the screenshot was a spreadsheet, a Claude conversation, or a YouTube video. Same size every time.

So 50 screenshots become 50 small vectors plus 50 short text descriptions plus 50 thumbs-up/thumbs-down signals. That's maybe 2MB total. Not growing out of control.

But here's where it gets smarter. Those 50 vectors cluster. Maybe 12 of them are "Claude conversations I liked." Maybe 8 are "spreadsheet layouts I preferred." Maybe 5 are "search results I found useful." The system doesn't need to remember all 50 individually — it learns the centroid of each cluster. The center of gravity of "things David likes in this category." New snapshots either reinforce an existing cluster (making the preference stronger) or start a new one (discovering a new preference category).

So the model doesn't grow linearly with snapshots. It grows with the number of distinct preference categories you have. You might take 10,000 snapshots over six months and end up with maybe 40-80 preference clusters. Fixed size. Stable.

Old snapshots that already contributed to a cluster can be archived or deleted — their signal is already baked into the centroid. You keep the knowledge, discard the raw data. This is literally how human memory works — you don't remember every meal, you remember what you like to eat.

The retrieval question — does it go hunting through folders?

No. The clusters live in memory as a lightweight index. When a new signal comes in (you're on a webpage, you opened a file, you're looking at search results), the system embeds that current context and does a nearest-neighbor lookup against your preference clusters. That's a millisecond operation even with hundreds of clusters. It doesn't open folders. It doesn't search. It just asks "which of David's known preferences is this closest to?" and gets an answer instantly.

The folders are for archival — raw snapshots, logs, the audit trail. The live model is a small in-memory structure. Think of it like: the folders are the filing cabinet, the live model is the cheat sheet pinned to the wall.

Now the architecture you're describing:

You're talking about three layers:

The capture layer — hotkey (Ctrl+Space or whatever), pulls up a small GUI, you can screenshot the current window, thumbs up/down, type a brief note. Maybe auto-detect the window and grab underlying data (if it's a browser, grab the URL and page content; if it's Excel, grab the file path and active sheet; if it's Claude, grab the conversation text). This is the BIL/AHK territory you've already started.

The processing layer — takes that capture, runs it through moondream or mistral locally, extracts a text description + embedding, applies your thumbs signal, updates the preference clusters. This runs on a schedule or triggered — every hour or two it processes the queue, or immediately on manual capture.

The application layer — the preference model sits as a service that anything can query. SearXNG asks "rank these results for David." The file organizer asks "where does David want this?" A new AI session asks "how does David like responses shaped?" The browser plugin asks "should I block this popup?" All hitting the same small, fast preference index.

The thing Codex is building on the BIL side is layers one and two. The dashboard you already have is the beginning of layer three's visibility. The hotkey GUI you're describing is a better capture interface for layer one.

The Chrome plugin is another input source. The hotkey GUI is another. They all feed the same preference model.

Alright, let me lay out the actual architecture for the hotkey capture system. This is the piece that turns BRAIN from passive observer into something you actively train.

The Hotkey Flow

You hit Ctrl+Space (or whatever key — we'll make it configurable). A small, minimal GUI pops up overlaying your current screen. It needs to do exactly five things:

First, it auto-identifies the active window — the app name, window title, and if it's a browser, the URL. This is trivial on Windows with pywin32 or AHK. So before you even do anything, the GUI already says something like "Claude — Preference Engine Architecture" or "pgAdmin — kj database" or "Obsidian — Law9_canonical.md."

Second, it takes a screenshot of that window. Not the whole desktop — just the focused window. Stored temporarily in a capture queue.

Third, you see three buttons: 👍 👎 and a text input field. Thumbs up means "more of this." Thumbs down means "less of this." The text field is optional — a brief note like "good response structure" or "hate this layout" or "save this query."

Fourth, there's a dropdown or auto-detected category. The system guesses based on context: "AI Conversation," "Code," "Document," "Search Results," "Spreadsheet," "File Manager," "Browser — Article." You can override it but usually won't need to.

Fifth, you hit Enter or click Submit. The GUI vanishes. Total interaction time: 2-4 seconds.

What Happens After You Dismiss the GUI

The capture goes into a local queue at D:\BIL\data\captures\. Each capture is a JSON file:

The processing pipeline picks this up on schedule (or immediately, your choice). It runs the screenshot through moondream on your NAS, which produces a text description of what's on screen. That description gets embedded into a vector. The vector, the signal (positive/negative), the category, and the note all get fed into the preference cluster engine.

The Category System — This Is Where Naming Matters

You said you get lost in naming, so let's lock this down now. The categories aren't folders. They're tags the system uses to route preferences to the right cluster family. Here's the proposed taxonomy:

ai_conversation — Any AI chat window (Claude, Gemini, GPT, Codex). Sub-learns: response style preferences, depth preferences, formatting preferences, which AI for which task.

code — VS Code, terminal, any IDE. Sub-learns: language preferences, comment style, file structure, naming conventions, debugging patterns.

research — Browser on academic sites, documentation, papers. Sub-learns: source preferences, reading depth, which domains you trust, search patterns.

writing — Obsidian, Word, any text editor when writing prose. Sub-learns: structure preferences, voice patterns, section ordering, formatting habits.

data — pgAdmin, spreadsheets, database tools. Sub-learns: query patterns, table layouts, column ordering, which databases for which tasks.

files — File Explorer, any file management. Sub-learns: folder structure, naming conventions, where things go, sort order.

media — YouTube, TikTok, audio, video editing. Sub-learns: content type preferences, completion rates, engagement patterns.

comms — Email, Slack, any communication tool. Sub-learns: tone per recipient, response speed, priority handling.

system — Settings, control panel, infrastructure management. Sub-learns: configuration preferences, tool choices, automation patterns.

browser_general — Browser on uncategorized sites. The catch-all that gets reclassified as the system learns.

The system auto-detects category from the window info. Chrome on docs.cloudflare.com → research. Chrome on claude.ai → ai_conversation. VS Code → code. It gets this right 80% of the time out of the box and improves as you correct it.

## How to query
### 1. Semantic search (Qdrant)
Ask in natural language. The vector store answers by similarity, not keyword match.
Good queries:
- "What did we decide about NotebookLM auth?"
- "Where did we leave the GTQ build folder structure?"
- "Show me sessions about Forge monorepo consolidation"
- "What threads are still open on the AI comms hub?"
Bad queries (too narrow for vector search — use Postgres):
- "Find session SESSION_2026-05-07_113830"
- "Sessions between April 1 and April 15"
### 2. Structured search (Postgres)
For dates, file counts, specific session IDs, or aggregations:
sql
SELECT session_id, processed_at, manifest_count, decision_count, thread_count
FROM session_handoffs
WHERE processed_at >= '2026-05-01'
ORDER BY processed_at DESC;

### 3. Combined workflow (recommended)
1. Vector search to find relevant sessions (returns 3-5 candidates)
2. Pull the matching *_summary.md files for full context
3. If you need raw conversation, look in the full-conversation mirror

The collection grows by 1+ session per day. Watch for:
- **Decision reversal** — same topic, opposite decision in a later session. Surface both.
